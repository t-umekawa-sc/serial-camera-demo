## 概要

スマートフォンのブラウザでカメラを起動し、画面上の赤いガイド枠に合わせた製造番号部分だけを切り出して撮影・蓄積し、最終的にサーバへ一括アップロードすることを想定したWebデモアプリケーション。

現段階は **通常Web版（フロントエンドのみ）** で、アップロード処理はダミー実装。サーバ側のOCR連携は未着手。

## 技術構成

- 純粋な静的Webアプリ（HTML / CSS / 素のJavaScript）。サーバ処理は不要
- ビルドツール・フレームワークは不使用
- バーコード読み取りエンジンは二段構え。対応端末（主にAndroid Chrome）では **ブラウザ標準の `BarcodeDetector` API**（OSのML Kit＝標準スキャナと同じネイティブ実装）を優先し、密なCODE 128等の読取精度を上げる。非対応端末（iOS Safari等）や未検出時は **ZXing（`@zxing/library`）** にフォールバック。ZXingはCDNから読み込む唯一の外部ライブラリ（BarcodeDetectorはブラウザ内蔵で追加ライブラリ不要）
- BarcodeDetectorは起動時に `getSupportedFormats()` で対応フォーマットを確認し、ZXingと同じ「読取種別／読取対象」の選択に合わせて対象フォーマットを絞って生成する（CODABAR・ITF除外やQRモードの区別はネイティブ経路でも維持）。HTTPS必須の制約はZXing経路と同じ
- カメラ操作は自前の `getUserMedia`（背面カメラ優先、2560×1440を要求＝密度の高いCODE 128等で細いバーに十分なピクセルを確保。実際の解像度は端末依存）。起動後、対応端末では連続オートフォーカス（`focusMode: continuous`）を `applyConstraints` で要求。映像をタップするとその位置にフォーカスを合わせる（`pointsOfInterest` ＋ `single-shot`／`continuous`、対応端末のみ。object-fit:cover を考慮して座標変換）。バーコードは「バーコード読取（長押し）」ボタンを押している間、ガイド枠内を約200ms間隔で連続解析（ZXing `MultiFormatReader`）。読めた時点で成功し停止、指を離すまでに読めなければエラー
- **読取種別の切り替え**: 「読取種別：バーコード／QR」トグルで、解析対象フォーマットを1次元バーコード（CODE_128/39/93・ITF・EAN・UPC・CODABAR）と2次元コード（QR・DataMatrix・Aztec・PDF417）に切り替える。ZXingはモードに含まれないフォーマットを解析しないため、バーコードとQRが近接印字されていても狙った方だけを読める（既定はバーコード）
- 各フレームは「素の画像 → 失敗時のみ影補正 → さらに失敗時は反転（ネガ）」の三段で解析。影補正は背景の明るさ分布（縮小描画でぼかして推定）で割って照明ムラを平坦化し、軽くコントラスト強調する処理（固定影に有効）。反転は黒地に白バー等の反転印字に対応。いずれも素画像を先に試すため、きれいなラベルでは速度・精度を落とさない
- 撮影とバーコード読み取りは同じガイド枠の切り出し座標（`getGuideSourceRect`）を共有
- 画像の切り出しは Canvas（`drawImage` + `toBlob`、JPEG品質0.9）
- `index.html` の `style.css` / `app.js` 参照にバージョン文字列（`?v=2`）を付与し、更新時のキャッシュ不一致を防止

## 実装済み機能

- **カメラ起動**: 起動時に自動でカメラを開始（ZXing経由）。失敗時はステータス表示とアラートで通知
- **ガイド枠内のバーコード読み取り（長押し連続試行）**: 「バーコード読取（長押し）」ボタンを押している間、赤いガイド枠内を連続で読み取り試行。影で一部フレームが失敗しても、押している間に読めたフレームで成功にできる。成功時は検出値と種別をバナー表示（対応端末ではバイブ通知）
- **読取種別の切り替え（バーコード／QR）**: トグルボタンで解析対象を1次元バーコードと2次元コード（QR等）に切り替え。近接印字されたQRに引っ張られず狙ったコードだけを読む狙い（既定はバーコード）。読み取り中は切り替え不可
- **読取対象バーコードの複数選択**: 「読取対象」パネルで1次元バーコード9種（CODE_128/39/93・CODABAR・ITF・EAN-13/8・UPC-A/E）をチェックボックスで個別ON/OFF。誤読しやすいCODABAR・ITFを外すなど用途に合わせて絞れる。**既定はCODABAR・ITFを外した7種**（誤読対策）。選択は `localStorage` に保持し次回起動時も維持。QRモード時は無効（QR側は固定フォーマット）
- **ネイティブ読取（BarcodeDetector）優先**: 対応端末では各フレームをまずブラウザ標準の `BarcodeDetector`（OSのML Kit）で解析。純JSのZXingより密なCODE 128等に強く、読取精度・速度が向上。未対応端末や未検出時のみZXingの三段解析に回すため、iPhone等でも従来通り動作
- **読取エンジンの表示**: 画面上部に現在有効なエンジン（「ネイティブ(高精度)」／「ZXing」）を常時バッジ表示。読み取り成功時は結果バナーにも、実際に読んだエンジン（ネイティブ/ZXing）を併記。実機で高精度エンジンが効いているかを確認できる
- **影補正の前処理**: 連続試行の各フレームで、素の画像で読めない場合に照明ムラ補正＋コントラスト強調をかけた画像で再解析。倉庫などの固定影に対する読み取り率を底上げ（テカリ・白飛び・ブレには効果なし）
- **反転（ネガ）画像の追試**: 影補正でも読めない場合、白黒を反転した画像でも解析。黒地に白バー等の反転印字に対応
- **読み取り失敗時のエラー表示**: 指を離すまでに枠内のバーコードを検出できなかった場合、同じバナーを赤色にして「読み取り失敗」と調整を促すメッセージを表示
- **タップフォーカス**: 映像をタップするとその位置にピントを合わせる（対応端末のみ）。タップ位置に黄色のフォーカス枠を一瞬表示。連続AFでボケる場合の合焦手段
- **ガイド枠撮影**: 画面の赤枠と同じ範囲を映像解像度に合わせて切り出し、メモリ上に保持
- **撮影ガイドの補助表示**: S/Nの先頭を合わせる開始マーカー（左端から内側にインセット）と、傾き防止の水平基準線を枠内に表示
- **撮り方のコツ表示**: 精度を上げる撮影のポイント（先頭/水平合わせ、近づく、ピント、明るさ・反射回避、正面撮影など）をパネルで表示。初回起動時に自動表示し、「コツ」ボタンでいつでも再表示可能
- **連続撮影**: 撮影枚数をカウント表示し、複数枚を蓄積
- **画像ID付与**: `YYYYMMDD_HHMMSS_連番` 形式のIDを自動生成
- **一覧確認パネル**: 撮影済み画像をサムネイル・撮影日時・IDで一覧表示
- **個別削除**: 一覧から画像を削除（`URL.revokeObjectURL` でメモリ解放）
- **画像の保存／共有**: 一覧の各画像に「保存」（`<a download>` で端末に `{ID}.jpg` 保存。Android向け）と「共有」（Web Share API `navigator.share({files})` で共有シートを開き、保存を挟まず Teams 等へ直接送信。iOS Safari向けの本命）を追加。共有非対応端末は保存にフォールバック。**実機サンプルをPCへ渡し `sn-reader` CLI の入力にする用途**
- **一括アップロード（ダミー）**: `FormData` を組み立てるがアラート表示のみ。実送信は未実装
- **画面向き対応**: 縦・横（landscape）でレイアウトを切り替え

## S/N 読み取りCLI（`sn-reader/` ・Docker）

Webアプリとは別に、ガイド枠内に切り出し済みの画像ファイルからS/Nバーコードを読み取るデモCLIを追加（まだWebアプリ化はせず、サーバOCRの前段検証）。

- `sn-reader/analyze.py`: ファイルパス（glob可）を渡すと **pyzbar(ZBar)** と **OpenCV BarcodeDetector** の両方で解析し、どちらがどの前処理で読めたかを並べて表示（`--json` で機械可読出力）
- 前処理は Webアプリと同思想の「素画像 → Otsu二値化 → 照明ムラ補正 → 反転」を上から順に試行
- **対象シンボロジーの限定**: pyzbar は既定で **CODE128/CODE39 のみ**を対象（`--symbols` で変更、`--all-symbols` で無制限）。Webアプリの既定（CODABAR・ITF除外）と統一。ITF(I25) は数字列の部分一致で誤読しやすいため既定で外す
- 実行環境は Docker（`sn-reader/Dockerfile`）。`samples/` を `/data` にマウントして解析。サンプル画像は実S/Nを含むため `.gitignore` でコミット対象外
- **エンジン比較の知見**: 対象の CODE 39/128 は **pyzbar が安定**。OpenCV BarcodeDetector は実質 EAN/UPC 専用で CODE 128 未検出・CODE 39 誤読があり本用途は不向き。実運用は pyzbar 軸の見込み（比較用に両方は残置）

### 動作確認済み

- 生成した CODE 128 / CODE 39 のテストバーコードで pyzbar が正しく読取（Docker ビルド・実行とも成功）
- 実機で撮影・Teams 経由でPCへ渡した実画像2枚で検証：
  - 商品の JAN(EAN13) `4902205021964` → 両エンジンとも読取OK（ただし製品コードでありS/Nではない）
  - 本命の `S/N: 10606540522247`（CODE 128）→ **両エンジンとも失敗**。原因は撮影品質（画面の再撮影によるモアレ、ピンボケ、低解像度 410×1152px、バーコード下端の見切れ）。アップスケール＋シャープ化でも復元不可。**CLIのパイプライン自体は正常で、入力画像側の問題**
- **画面再撮影でない元写真（2000×1500）でも再検証**（PowerShell/System.Drawing で S/N 部分を枠内に切り出し `sn_buffalo_hd-le2u3_v1.jpg`）：
  - **ITF 誤読を発見**: 無制限デコードだと本来の `10606540522247` ではなく ITF(I25) として `40522247`（`rect=0x0` の縮退偽陽性）を拾う。→ **pyzbar を CODE128/CODE39 に限定する修正を実施**（Webアプリ既定と統一。誤った番号を出さず「検出なし」と正しく報告）
  - **限定後は S/N が読めない**: 拡大(2〜6倍)×シャープ化×Otsu/適応二値化×傾き補正(±3°) を総当たりしても全滅。**ラベル全体**で試しても ZBar が読めたのは QR 1個（`http://86886.jp/recovery`）のみで、S/Nバーコード・脇のQR・左下CODE39は全て失敗
  - 根本原因: 元写真が **3MP でバーコード幅わずか約590px＋ソフトフォーカス**。画面再撮影でなくても密なリニアバーコードには解像度・鮮鋭度が不足
- 結論: **鮮明な現物ラベルの撮り直しが必要**（ピント・接写・バーコード全体＋左右余白を枠内に収める。写真内でバーコード幅 1000px 以上が目安）

## OCR 読み取りCLI（`ocr-reader/` ・Docker / PaddleOCR）

バーコードが読めないラベルでも、印字テキストなら読める場合があるため、`sn-reader`（バーコード）の**補完**として OCR デモを追加。

- `ocr-reader/ocr.py`: PaddleOCR（`lang=japan`：日本語＋英数字）で画像内の全テキスト行を信頼度つきで検出・認識し、`S/N`ラベル直後の英数字／10桁以上の連続数字から **S/N候補**を抽出（`--json` あり）
- 実行環境は Docker（`ocr-reader/Dockerfile`）。リポジトリ直下 `samples/` を `/data` にマウントして解析
- **ビルド時の難所と対処**（この環境固有の落とし穴。`Dockerfile`/`fetch_models.py` に対策済み）:
  - `paddlepaddle==2.6.2` は **numpy 1.x ABI** 前提 → `numpy==1.26.4` にピン
  - 巨大依存を**一括**pip installするとコンパイル済み拡張(.so)が壊れ `zlib.error: inconsistent stream state` → **段階的に別レイヤで導入**＋Cython/pyclipperをクリーン再インストール
  - paddle を import した状態で tar 展開すると **zlib シンボル衝突で segfault**（`inflateReset2`）→ モデルは **paddle 非importのクリーンなプロセスで事前DL＋展開**（`fetch_models.py`）。以後 PaddleOCR() は展開を走らせず、推論も zlib 不使用のため安全。**実行時ネットワーク不要**

### 動作確認済み（重要な知見）

- **切り出し済み S/N 画像で `10606540522247` を信頼度1.00で正読**。**バーコードでは同一画像から全く読めなかった S/N が、OCR なら取得できた**（バーコードとOCRは相補的）
- **全体写真そのままは精度が低い**（ラベルが写真の一部で小さく、テキスト検出が拾えず誤検出になる。例: 全景の image(2) では S/N を検出できず）。→ **アプリのガイド枠で切り出した領域を入力にすると精度が跳ね上がる**（設計方針と一致）
- JAN 数字は概ね読取（先頭桁のみ誤読）、`Made in Japan` は軽微な誤読あり。**数字主体のS/Nはとくに好成績**

## 留意点

- アップロードはダミーのため、撮影データは外部に送信されない
- 画像はすべてブラウザのメモリ上にのみ存在し、タブを閉じる／リロードで失われる
- カメラ権限・HTTPS要件があるため、`file://` 直開きやHTTP接続では動作しないことがある
- ガイド枠はS/Nの長さが不明な前提で広め（幅80%）に取り、クロップは枠全体を切り出す（右端の欠けを防ぐ）。開始マーカーは左端に密着させず、OCR用の余白を確保するためインセットしている。実機で過不足が分かり次第、枠幅の調整が望ましい
